## 禁止的行为

### ❌ 错误的做法

| 错误做法 | 问题 | 正确做法 |
|---------|------|----------|
| 仅阅读代码就确定需要修改 | 可能误判，实际功能正常 | 先测试确认问题 |
| 假设服务器行为 | 服务器可能与预期不同 | 用真实请求验证 |
| 基于理论分析决策 | 理论≠实际 | 用测试数据说话 |
| 盲目追求代码一致 | 实现可以不同，行为一致即可 | 关注功能等价 |
| 忽略边缘情况测试 | 边缘情况容易出问题 | 全面测试各种场景 |

## 测试要求

### 必须测试的场景

#### 1. 基础功能测试

| 功能 | 测试内容 | 验证点 |
|------|---------|--------|
| DID 创建 | 创建身份 | DID 格式、文档结构、Proof |
| DID 注册 | 注册到 awiki.ai | 注册成功、user_id 返回 |
| JWT 获取 | 签名验证 | JWT token 获取成功 |
| 明文消息 | 发送/接收 | 消息内容一致 |
| E2EE 通信 | 握手/加密/解密 | 消息能互相解密 |

#### 2. 互操作性测试

| 测试场景 | Python → Node.js | Node.js → Python |
|---------|-----------------|-----------------|
| 明文消息 | ✅ 必须通过 | ✅ 必须通过 |
| E2EE 握手 | ✅ 必须通过 | ✅ 必须通过 |
| E2EE 消息 | ✅ 必须通过 | ✅ 必须通过 |

#### 3. 边界情况测试

| 场景 | 测试内容 |
|------|---------|
| 网络错误 | 超时、断网、服务器错误 |
| 无效输入 | 错误格式、缺失字段 |
| 并发请求 | 同时多个请求 |
| 大消息 | 超过正常大小的消息 |
| 特殊字符 | 中文、emoji、控制字符 |

## 测试环境

### 测试配置


### 测试数据记录

每次测试必须记录：

1. **测试时间**: ISO 8601 格式
2. **测试环境**: awiki.ai 或其他
3. **测试版本**: Python 版本、Node.js 版本
4. **测试输入**: 请求参数、输入数据
5. **实际输出**: 响应内容、返回数据
6. **预期输出**: 期望的结果
7. **测试结果**: 通过/失败
8. **问题描述**: 失败原因分析

## 升级决策流程

### 决策标准

| 问题类型 | 决策依据 | 优先级 |
|---------|---------|--------|
| 功能失败 | awiki.ai 测试失败 | 🔴 高 |
| 互操作失败 | Python↔Node.js 无法互通 | 🔴 高 |
| 行为差异 | 与 Python 输出不一致但功能正常 | 🟡 中 |
| 代码风格 | 仅代码组织不同 | 🟢 低 |
| 理论问题 | 仅代码阅读发现的问题 | 🟢 低 |

### 决策流程

```
1. 发现问题
   ↓
2. 记录问题现象
   ↓
3. 在 awiki.ai 复现问题
   ↓
4. 确认是否为真实问题（非假设）
   ↓
5. 评估影响范围
   ↓
6. 制定修复方案
   ↓
7. 实施修复
   ↓
8. 在 awiki.ai 验证修复
   ↓
9. 记录修复过程
```

## 示例：正确的升级流程

### 场景：发现签名格式差异

#### ❌ 错误做法

```
1. 阅读代码发现 Python 用 DER 格式，Node.js 用 R||S 格式
2. 立即修改 Node.js 使用 DER 格式
3. 提交代码
```

#### ✅ 正确做法

```
1. 阅读代码发现格式差异
2. 记录差异但不立即修改
3. 在 awiki.ai 测试 Node.js 注册功能
   - 如果成功：说明服务器接受 R||S 格式，无需修改
   - 如果失败：继续下一步
4. 修改 Node.js 使用 DER 格式
5. 在 awiki.ai 验证修改后是否成功
6. 记录测试结果和修复过程
```

## 示例：测试驱动的问题确认

### 场景：E2EE 消息无法解密

#### 测试步骤

```bash
# 1. Python 发起 E2EE 握手
cd python-client/scripts
python e2ee_messaging.py --init --peer <DID>

# 2. Node.js 响应握手
cd nodejs-client
node scripts/e2ee_messaging.js --accept --peer <DID>

# 3. Python 发送加密消息
python e2ee_messaging.py --send --peer <DID> --content "Hello"

# 4. Node.js 尝试解密
node scripts/e2ee_messaging.js --recv --peer <DID>

# 5. 记录结果
# 成功：输出 "Hello"
# 失败：记录错误信息
```

#### 结果分析

```
如果失败：
1. 检查 Ratchet 派生是否一致
2. 检查 HPKE 参数是否一致
3. 检查序列号处理是否一致
4. 修改后重新测试
```
