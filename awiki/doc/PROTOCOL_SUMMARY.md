## 1. 问题背景

Python 版本 (awiki-agent-id-message) 和 awiki.ai 网站的服务端是同一个团队开发的。这两个项目之间的接口和时序、协议，有大量实现细节没有记录到任何文档，而是他们团队在开发过程直接口头交流确定的。

现在我们没有 awiki.ai 网站服务端的代码，需要把 Python 版本移植到 Node.js。

## 2. 分析结果

### 2.1 识别的隐性协议

**共识别 5 个主要隐性协议**：

| 协议 | 说明 | 关键文件 | 风险等级 |
|------|------|----------|---------|
| **DID WBA 认证协议** | 基于 secp256k1 的签名认证，JWT 令牌管理 | `auth.py` | 🟡 中 |
| **JSON-RPC 2.0 调用协议** | 标准 RPC 格式 + 401 自动重试 | `rpc.py` | 🟡 中 |
| **E2EE 端到端加密协议** | HPKE (RFC 9180) + Chain Ratchet，三密钥体系 | `e2ee.py` | 🔴 高 |
| **WebSocket 推送协议** | JWT 查询参数认证，推送通知格式 | `ws.py` | 🟡 中 |
| **Handle 注册协议** | OTP 验证，电话号码规范化 | `handle.py` | 🟢 低 |

### 2.2 主要风险点

| 风险等级 | 风险项 | 描述 | 克服方案 |
|----------|--------|------|----------|
| 🔴 高 | E2EE 版本硬编码 | 版本 "1.1" 硬编码，服务端升级可能导致不兼容 | 实现版本协商机制 |
| 🔴 高 | JWT 刷新逻辑 | 依赖 401 被动触发，高延迟网络可能失败 | 实现主动刷新（过期前） |
| 🔴 高 | HPKE 状态持久化 | 复杂的会话状态序列化/反序列化 | 严格测试状态序列化 |
| 🟡 中 | 未文档化错误码 | -32003、-32004 等错误码含义不明 | 实现错误码映射表 |
| 🟡 中 | URL 路径硬编码 | RPC 端点路径写死在代码中 | 配置化端点路径 |

## 6. 关键发现

### 6.1 隐性协议细节

#### DID WBA 认证
- 认证头格式：`DIDWba {did}:{signature}:{timestamp}`
- JWT 从响应头 `Authorization: Bearer {token}` 或响应体 `access_token` 提取
- 401 响应时自动清除过期令牌并重新获取

#### JSON-RPC 调用
- 错误码 `-32003`: DID already registered
- 错误码 `-32004`: Slug already taken
- 收到 401 时自动重试一次

#### E2EE 加密
- 版本硬编码为 "1.1"
- 三密钥体系：key-1(secp256k1), key-2(secp256r1), key-3(X25519)
- 会话过期时间：86400 秒 (24 小时)
- 序列号最大跳过：256

#### WebSocket 推送
- JWT 通过查询参数 `?token={jwt}` 传递
- 推送通知无 `id` 字段
- URL 自动从 HTTP 转换为 WebSocket

#### Handle 注册
- 中国号码自动添加 `+86` 前缀
- OTP 代码自动去除空白字符
- 短 Handle (≤4 字符) 需要邀请码

### 6.2 魔法值汇总

```javascript
// URL 路径
HANDLE_RPC = "/user-service/handle/rpc"
DID_AUTH_RPC = "/user-service/did-auth/rpc"
MESSAGE_RPC = "/message/rpc"

// 版本号
SUPPORTED_E2EE_VERSION = "1.1"
_STATE_VERSION = "hpke_v1"

// 时间限制
DEFAULT_EXPIRES = 86400  // 24 小时
MAX_SKIP = 256
DEFAULT_TIMEOUT = 30.0

// 错误码
ERROR_DID_REGISTERED = -32003
ERROR_SLUG_TAKEN = -32004
```
